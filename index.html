01
Установка Ubuntu / WSL и первые команды
✓
▶

Linux — операционная система большинства серверов в мире. Ubuntu — самый популярный дистрибутив для разработчиков. WSL (Windows Subsystem for Linux) позволяет запустить Linux прямо в Windows 10/11 без виртуальной машины.

# В PowerShell от администратора: wsl --install # установит Ubuntu по умолчанию wsl --install -d Ubuntu-22.04 # конкретная версия wsl --list --verbose # список установленных # Первые команды после входа: $ whoami # кто ты? $ hostname # имя машины $ uname -a # версия ядра Linux $ echo $HOME # домашняя папка $ echo $SHELL # твоя оболочка (bash/zsh) $ date # текущая дата и время $ uptime # сколько работает система $ history # история команд
💡 Структура команды

команда -опции аргументы. Например: ls -la /home — команда ls, опции -l (длинный формат) и -a (скрытые файлы), аргумент /home. Опции можно объединять: -la = -l -a.

Задача дня

02
Навигация: ls, cd, pwd, tree
✓
▶

В Linux всё — файлы и папки. pwd (Print Working Directory) показывает где ты. ls — список файлов. cd — смена директории. Абсолютный путь начинается с /, относительный — с текущей папки.

$ pwd # /home/alisher $ ls # файлы в текущей папке $ ls -l # подробный список $ ls -la # + скрытые файлы (начинаются с .) $ ls -lh # размеры в читаемом виде (KB, MB) $ ls /etc # содержимое другой папки # Навигация $ cd /etc # перейти по абсолютному пути $ cd .. # на уровень выше $ cd ~ # в домашнюю папку $ cd - # в предыдущую папку $ cd ~/projects/app # относительно домашней # Дерево папок $ sudo apt install tree $ tree -L 2 ~ # дерево глубиной 2 уровня
drwxr-xr-x 5 alisher alisher 4096 May 1 12:00 projects -rw-r--r-- 1 alisher alisher 220 May 1 09:00 .bashrc ↑тип+права ↑links ↑owner ↑group ↑size ↑дата ↑имя d = папка, - = файл, l = символическая ссылка
Задача дня

Пройди путь: / → /etc → /var → /var/log → домашняя папка — используя только cd. В каждой папке выполни ls -la. Установи tree и посмотри структуру своей домашней папки на 3 уровня.

05
Редактор nano: основы работы
✓
▶

nano — простой консольный текстовый редактор. Всегда доступен на серверах. Управление через Ctrl+буква (показано внизу экрана). Незаменим для быстрого редактирования конфигов.

$ nano filename.txt # открыть/создать файл # Горячие клавиши внутри nano: # Ctrl+O → сохранить (Write Out) # Ctrl+X → выйти # Ctrl+K → вырезать строку # Ctrl+U → вставить # Ctrl+W → поиск (Where is) # Ctrl+\ → замена # Ctrl+G → справка # Alt+U → отменить действие # Создание bash-скрипта через nano $ nano hello.sh # Внутри напиши: #!/bin/bash echo "Привет, $USER!" echo "Сегодня: $(date)" echo "Директория: $(pwd)" # Ctrl+O → Enter → Ctrl+X $ chmod +x hello.sh && ./hello.sh

07
Переменные окружения и .bashrc
✓
▶

Переменные окружения — это именованные значения доступные всем процессам в системе. PATH определяет где Bash ищет команды. .bashrc и .bash_profile — файлы инициализации, выполняемые при старте оболочки. Настройка этих файлов делает работу в терминале намного удобнее.

# Просмотр переменных окружения $ env # все переменные $ printenv PATH # значение конкретной $ echo $HOME # домашняя папка $ echo $USER # текущий пользователь $ echo $PATH # пути поиска команд # Установка переменных $ MY_VAR="hello" # локальная (только текущий shell) $ export MY_VAR="hello" # экспортировать в дочерние процессы $ export DB_HOST="localhost" # типичный пример для приложений # .bashrc — конфигурация интерактивного shell $ nano ~/.bashrc # Добавь в .bashrc: alias ll='ls -la' alias gs='git status' alias py='python3' export PATH="$HOME/.local/bin:$PATH" # Применить изменения без перезапуска $ source ~/.bashrc # или: . ~/.bashrc # Посмотреть все алиасы $ alias
💡 .env файлы в разработке

В реальных проектах секреты (API ключи, пароли) хранят в файле .env который никогда не коммитится в Git. Приложение читает их через переменные окружения. Пример: DATABASE_URL=postgres://user:pass@localhost/db

Задача дня

Создай 3 полезных алиаса в ~/.bashrc: для быстрого перехода в папку Projects, для просмотра IP-адреса, для обновления системы. Добавь свой путь в PATH. Создай файл .env с переменными DB_HOST, DB_PORT, APP_SECRET и загрузи их через source .env. Проверь через echo.

09
grep: поиск текста как профессионал
✓
▶

grep (Global Regular Expression Print) — мощнейший инструмент поиска текста. Ищет по строкам, поддерживает регулярные выражения. Незаменим при анализе логов и отладке.

# Базовый поиск $ grep "error" /var/log/syslog # найти строки с "error" $ grep -i "error" app.log # -i игнорировать регистр $ grep -n "error" app.log # -n с номерами строк $ grep -v "DEBUG" app.log # -v исключить строки с DEBUG $ grep -c "ERROR" app.log # -c подсчитать количество # Контекст вокруг найденного $ grep -B 2 -A 3 "Exception" app.log # 2 строки до и 3 после # Рекурсивный поиск в папке $ grep -r "password" /etc/ # ищет во всех файлах /etc $ grep -r "API_KEY" ~/projects/ # найти ключи в коде (опасно!) # Регулярные выражения $ grep -E "ERROR|CRITICAL" app.log # OR через | $ grep -E "[0-9]{1,3}\.[0-9]{1,3}" access.log # IP-адреса # Комбо с конвейером $ cat /var/log/auth.log | grep "Failed" | grep "root" $ ps aux | grep python
Задача дня

find: поиск файлов по всем параметрам
✓
▶
# Синтаксис: find [где] [условия] [действие] # По имени $ find /home -name "*.py" # все Python-файлы $ find . -name "config*" # файлы начинающиеся с config $ find / -iname "readme.md" # без учёта регистра # По типу $ find /etc -type f # только файлы $ find /home -type d # только директории # По размеру $ find / -size +100M # больше 100MB $ find /var/log -size +10M # большие логи # По времени изменения $ find / -mtime -1 # изменён за последние 24ч $ find /etc -newer /etc/passwd # новее чем passwd # По правам (важно для ИБ!) $ find / -perm 777 # файлы с опасными правами $ find / -perm -4000 # SUID-файлы (могут быть опасны) # Выполнить команду для найденных $ find /tmp -name "*.log" -delete # найти и удалить $ find . -name "*.py" -exec wc -l {} \; # подсчитать строки в каждом

✓

14
awk и sed: мощная обработка текста
✓
▶

awk и sed — мощные инструменты обработки текста. sed (stream editor) делает замены и трансформации. awk обрабатывает структурированные данные по колонкам. В связке с grep и pipes — незаменимы для анализа логов и парсинга данных.

# sed — замена в файлах и потоках $ sed 's/old/new/' file.txt # заменить первое вхождение в строке $ sed 's/old/new/g' file.txt # заменить все вхождения (g = global) $ sed -i 's/localhost/127.0.0.1/g' config.txt # изменить файл напрямую $ sed -n '5,10p' file.txt # вывести строки 5-10 $ sed '/^#/d' config.txt # удалить строки с комментариями $ sed 's/ERROR/[ERROR]/g' app.log # выделить ошибки в логах # awk — обработка по полям # $1, $2... — поля (разделитель пробел/tab по умолчанию) $ awk '{print $1}' access.log # первое поле (IP в nginx-логе) $ awk '{print $1, $7}' access.log # IP и запрошенный URL $ awk -F: '{print $1, $3}' /etc/passwd # разделитель :, имя и UID # awk с условиями $ awk '$9 == "404" {print $1, $7}' access.log # только 404 ошибки $ awk '$5 > 1000000 {print $0}' data.txt # строки где поле 5 > 1MB # awk итог по колонке $ awk '{sum += $10} END {print "Total:", sum}' access.log # суммарный трафик # Практическая связка: топ IP-атакующих $ awk '/Failed/{print $(NF-3)}' /var/log/auth.log | sort | uniq -c | sort -rn | head -5
Задача дня

Используй sed чтобы заменить все вхождения своего имени в файле. Используй awk для анализа /etc/passwd: выведи только имена пользователей с UID больше 1000 (реальные пользователи). Напиши однострочник который считает суммарный размер запросов из nginx access.log.

Sudo: безопасные права суперпользователя
✓
▶

sudo (Substitute User DO) — выполнение команд с правами другого пользователя (обычно root). Безопаснее чем работать постоянно под root: логирует все команды, требует пароль, можно тонко настраивать через sudoers.

$ sudo command # выполнить как root $ sudo -i # войти в оболочку root $ sudo -l # что мне разрешено делать через sudo $ sudo !! # повторить последнюю команду с sudo # Настройка sudo: /etc/sudoers (редактировать только через visudo!) $ sudo visudo # Примеры строк в sudoers: alisher ALL=(ALL:ALL) ALL # полные права deploy ALL=(ALL) NOPASSWD: /bin/systemctl restart nginx # без пароля %developers ALL=(ALL) /usr/bin/apt # группа может использовать apt # Где хранятся sudo-логи $ grep sudo /var/log/auth.log | tail -20
⚠ Безопасность sudo

Никогда не давай NOPASSWD: ALL обычным пользователям. На сервере давай только минимально необходимые права (принцип наименьших привилегий). Регулярно проверяй sudoers: sudo cat /etc/sudoers.

Задача дня

Выполни sudo -l — посмотри что тебе разрешено. Обнови список пакетов через sudo apt update. Проверь лог sudo-действий в /var/log/auth.log. Объясни: почему лучше использовать sudo чем работать постоянно под root?

17
chmod: права доступа к файлам
✓
▶

Каждый файл имеет три группы прав: владелец (u), группа (g), остальные (o). Три типа прав: r(4)=чтение, w(2)=запись, x(1)=выполнение. chmod 644 = rw-r--r-- (владелец читает/пишет, остальные только читают).

Читаем права: -rw-r--r-- - = тип: файл (-), папка (d), ссылка (l) rw- = владелец: r=4 + w=2 + -=0 → 6 r-- = группа: r=4 + -=0 + -=0 → 4 r-- = остальные: r=4 + -=0 + -=0 → 4 Итого: 644
# Числовой способ $ chmod 644 file.txt # rw-r--r-- (обычный файл) $ chmod 755 script.sh # rwxr-xr-x (исполняемый скрипт) $ chmod 600 secret.env # rw------- (только владелец!) $ chmod 700 private_dir/ # rwx------ (только владелец) $ chmod 777 public_dir/ # ⚠ ВСЕ могут всё — ОПАСНО! # Символьный способ $ chmod +x script.sh # добавить выполнение $ chmod go-w file.txt # убрать запись у группы и остальных $ chmod u+x,g-w script.sh # комбо # Рекурсивно для папки $ chmod -R 755 /var/www/html/ # все файлы веб-сервера
ПраваЧислаПрименение
rw-------600Приватный ключ SSH, .env файлы
rw-r--r--644Обычные файлы, веб-страницы
rwxr-xr-x755Скрипты, исполняемые файлы, папки
rwx------700Папки с конфиденциальными данными
rwxrwxrwx777⚠ Никогда в продакшне!
Задача дня

Создай файлы и установи права: secret.env → 600, app.py → 644, deploy.sh → 755, private/ → 700. Проверь командой ls -la. Попробуй прочитать secret.env под другим пользователем — убедись что доступ запрещён.

SSH: безопасное удалённое подключение
✓
▶

SSH (Secure Shell) — стандартный протокол удалённого управления серверами. Ключевая пара (публичный + приватный ключ) безопаснее паролей. Знание SSH обязательно для любого backend-разработчика.

# Подключение по SSH $ ssh user@hostname # подключиться (спросит пароль) $ ssh user@192.168.1.100 # по IP $ ssh -p 2222 user@host # нестандартный порт # Генерация SSH-ключей $ ssh-keygen -t ed25519 -C "email@example.com" # Создаст: ~/.ssh/id_ed25519 (приватный) и ~/.ssh/id_ed25519.pub (публичный) # Копирование публичного ключа на сервер $ ssh-copy-id user@server # автоматически # Или вручную: добавить ~/.ssh/id_ed25519.pub в ~/.ssh/authorized_keys на сервере # Подключение по ключу (без пароля) $ ssh -i ~/.ssh/id_ed25519 user@server # Конфиг для удобства (~/.ssh/config) Host myserver HostName 192.168.1.100 User alisher IdentityFile ~/.ssh/id_ed25519 Port 22 $ ssh myserver # теперь так просто! # Передача файлов $ scp file.txt user@server:/tmp/ # скопировать на сервер $ scp -r ./app/ user@server:~/ # папку целиком
🔒 Безопасность SSH

Никогда не передавай приватный ключ (id_ed25519) — только публичный (.pub). Права на приватный ключ должны быть 600: chmod 600 ~/.ssh/id_ed25519. Иначе SSH откажется его использовать.

Задача дня

Сгенерируй пару SSH-ключей (ed25519). Изучи содержимое публичного ключа. Настрой ~/.ssh/config с псевдонимом для подключения. Если есть доступ к серверу — подключись по ключу. Проверь права папки ~/.ssh (должны быть 700) и файлов ключей (600).

20
Firewall: ufw и управление сетью
✓
▶

UFW (Uncomplicated Firewall) — простой интерфейс к iptables. На любом продакшн-сервере брандмауэр обязателен. Правило: по умолчанию всё запрещено, разрешаем только нужные порты.

# Статус и включение $ sudo ufw status # текущие правила $ sudo ufw status verbose # подробно $ sudo ufw enable # включить (сохраняется после перезагрузки) $ sudo ufw disable # отключить # Правила по умолчанию (делай это ПЕРВЫМ) $ sudo ufw default deny incoming # запретить всё входящее $ sudo ufw default allow outgoing # разрешить исходящее # Разрешить сервисы $ sudo ufw allow ssh # порт 22 (ВАЖНО: до enable!) $ sudo ufw allow 80 # HTTP $ sudo ufw allow 443 # HTTPS $ sudo ufw allow 5432 # PostgreSQL # Ограничить brute-force SSH $ sudo ufw limit ssh # максимум 6 попыток за 30 секунд # Запретить конкретный IP $ sudo ufw deny from 192.168.1.50 $ sudo ufw delete deny from 192.168.1.50 # удалить правило # Сетевые утилиты $ ss -tlnp # открытые порты и процессы $ netstat -tlnp # аналог (старый) $ curl ifconfig.me # внешний IP сервера
⚠ Важно

Всегда разрешай SSH (sudo ufw allow ssh) ПЕРЕД включением firewall (sudo ufw enable). Иначе заблокируешь себя и потеряешь доступ к серверу.

Задача дня

Включи UFW с правилами: запретить всё входящее, разрешить SSH, HTTP, HTTPS. Включи лимит на SSH для защиты от brute-force. Проверь открытые порты через ss -tlnp. Посмотри статус firewall. Если запускаешь веб-сервер — проверь доступность через браузер.

fail2ban и защита от атак
✓
▶

fail2ban — утилита которая автоматически блокирует IP-адреса при обнаружении подозрительной активности (брутфорс SSH, спам в веб-формах). Анализирует логи и добавляет правила в iptables/ufw.

# Установка $ sudo apt install fail2ban # Статус $ sudo systemctl status fail2ban $ sudo fail2ban-client status # список активных jail $ sudo fail2ban-client status sshd # статус jail для SSH # Конфигурация /etc/fail2ban/jail.local [DEFAULT] bantime = 3600 # блокировка на 1 час findtime = 600 # за 10 минут maxretry = 5 # максимум 5 попыток [sshd] enabled = true port = ssh logpath = /var/log/auth.log maxretry = 3 # строже для SSH # Управление $ sudo fail2ban-client set sshd unbanip 1.2.3.4 # разблокировать IP $ sudo fail2ban-client set sshd banip 5.6.7.8 # заблокировать вручную # Просмотр заблокированных IP $ sudo iptables -n -L INPUT | grep DROP
💡 Реальная защита сервера

Связка UFW + fail2ban + SSH по ключам закрывает 99% атак на типичный VPS. Дополнительно: смени порт SSH (не 22), отключи вход под root (PermitRootLogin no в /etc/ssh/sshd_config).

Задача дня

Установи и настрой fail2ban. Создай /etc/fail2ban/jail.local с настройками для SSH (maxretry=3, bantime=1h). Перезапусти сервис и проверь статус. Посмотри сколько IP уже заблокировано. Проверь логи fail2ban в /var/log/fail2ban.log.

Настройка безопасного окружения сервера

Настрой систему прав как в реальном продакшн-окружении.

  • Создать пользователя deploy без прав sudo (только для деплоя)
  • Папка /var/www/app: права 755, владелец www-data, группа developers
  • Папка /etc/app/secrets: права 700, только root читает
  • Файл .env: права 600, владелец deploy
  • Дать пользователю deploy право только на systemctl restart app через sudoers

Docker: основы контейнеризации
✓
▶

Docker позволяет упаковать приложение со всеми зависимостями в контейнер — изолированное окружение. Контейнеры запускаются одинаково на любой машине: ноутбук разработчика, CI/CD, продакшн-сервер. Это решает проблему «у меня работает, а у тебя нет».

# Установка Docker $ sudo apt install docker.io $ sudo usermod -aG docker $USER # добавить себя в группу (без sudo) $ newgrp docker # применить без перелогина # Основные команды $ docker version # версия $ docker pull ubuntu:22.04 # скачать образ $ docker images # список образов $ docker ps # запущенные контейнеры $ docker ps -a # все (включая остановленные) # Запуск контейнеров $ docker run ubuntu echo "hello" # запустить и выйти $ docker run -it ubuntu bash # интерактивный shell $ docker run -d -p 80:80 nginx # nginx в фоне, порт 80 # Управление $ docker stop container_id # остановить $ docker rm container_id # удалить контейнер $ docker rmi nginx # удалить образ $ docker logs container_id # логи контейнера $ docker exec -it container bash # войти в работающий

Мониторинг системы: htop, iotop, nethogs
✓
▶

Мониторинг — основа эксплуатации серверов. Нужно уметь быстро найти что жрёт CPU, память, диск или сеть. Это критично при инцидентах в продакшн.

# htop — интерактивный менеджер процессов $ sudo apt install htop $ htop # F5 = дерево процессов, F6 = сортировка, F9 = kill, q = выход # Использование памяти $ free -h # RAM и swap $ vmstat 1 5 # статистика каждую секунду, 5 раз # Диск $ df -h # свободное место на разделах $ du -sh /var/log/* # размер каждого лога $ du -sh * | sort -hr | head -10 # топ-10 самых больших папок $ sudo apt install iotop $ sudo iotop # какой процесс читает/пишет диск # Сеть $ sudo apt install nethogs $ sudo nethogs # трафик по процессам $ sudo apt install nload $ sudo nload # входящий/исходящий трафик # Быстрая диагностика одной командой $ top -bn1 | grep "Cpu" # CPU прямо сейчас $ ps aux --sort=-%cpu | head -5 # топ-5 по CPU $ ps aux --sort=-%mem | head -5 # топ-5 по памяти
💡 Load Average

Первые 3 числа в top и uptime — load average за 1, 5, 15 минут. Нормальное значение ≤ количества ядер CPU. Если число больше числа ядер — система перегружена. Узнай количество ядер: nproc.
